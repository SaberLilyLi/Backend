## 智能个人知识库助手 - 后端需求与实现说明（企业级版）

本文件从“产品 / 架构 / 实施”三个视角描述后端服务的需求，作为前后端联调和后续演进的基线文档。

---

### 一、项目背景与目标

- **项目名称**：智能个人知识库助手（后端服务）
- **技术栈**：Node.js (Express) + MongoDB + JWT 认证
- **系统角色**：
  - 终端用户（个人开发者、学生、职场人士）
  - 前端 Web 应用（Vue3 + TS）
  - 本后端服务（API 层，含 AI 能力集成）

- **业务目标**：
  - 为个人用户提供一套“可自管理、可搜索、可分析”的知识库后端服务。
  - 支撑前端实现：文档管理、智能检索、知识图谱、统计可视化等功能。
  - 在设计上符合企业级实践（分层架构、环境隔离、版本管理、统一错误码等），可作为简历项目展示工程能力。

- **非业务性目标**：
  - 提供清晰的 API 版本管理与环境区分能力（/api/{stage?}/v1）。
  - 保持代码结构可扩展（支持未来团队协作、多模态输入等新能力）。

---

### 二、范围说明

- **在本版本范围内的功能**：
  - 用户注册 / 登录 / 登出（单用户空间隔离）。
  - 文档与笔记的 CRUD，标签与分类管理。
  - 文档文件上传与内容解析（接口设计已预留，解析逻辑可按阶段实现）。
  - 语义搜索 & 简单问答（基于伪向量的 RAG 占位实现）。
  - 知识图谱数据输出（文档-标签-分类关系）。
  - 统计分析接口（概览、趋势）。

- **当前版本明确不覆盖，但需预留扩展空间的功能**：
  - 多租户 / 团队协作 / 细粒度权限体系（项目级、空间级权限）。
  - 语音输入、图片 OCR、多模态内容处理。
  - 多模型管理（不同大模型 / 向量引擎的路由与切换）。
  - 完整版本控制（差异对比、标签版本、回滚策略优化）。
  - 大规模数据性能优化（分片、缓存、多副本等）。

---

### 三、环境与版本管理要求

- **环境类型**：
  - 本地开发环境：developer 本地运行。
  - 联调 / 测试环境：与前端联调、QA 测试使用（如 q / n / o2 环境）。
  - 生产环境：对真实用户开放。

- **API 路径规范**：
  - 基础前缀：`/api`
  - 环境阶段（可选）：`/{stage}`，如 `/q`、`/n`、`/o2`。
  - API 版本：`/v1`（当前版本），未来可变更为 `v2`。
  - 组合规则：`/api/{stage?}/{version?}`。

- **版本兼容策略**：
  - 当前版本为 `v1`：
    - 推荐路径：`/api/v1/...`、`/api/q/v1/...`
    - 兼容路径：`/api/...`、`/api/q/...`（指向 v1）
  - 当前版本为 `v2`：
    - 推荐路径：`/api/v2/...`、`/api/q/v2/...`
    - 兼容路径：`/api/...`、`/api/q/...`（指向 v2）
    - 不再暴露 `/api/v1/...`。

### 四、核心数据模型设计（MongoDB）

> 以下仅列出与本项目相关的主要集合，字段略去部分内部实现。

-- **User**

- `username`：用户名（唯一，校验格式）。
- `email`：邮箱（唯一）。
- `password`：哈希后的密码。
- `avatarUrl`：头像地址（可选）。
- `role`：用户角色，`viewer` / `user` / `admin`，默认为 `user`。
- `createdAt`：创建时间。

-- **Document**

- `title`：文档标题。
- `content`：文档内容（Markdown / 纯文本等）。
- `content_type`：内容类型（默认 `markdown`）。
- `category`：分类名称（与 Category 逻辑关联）。
- `tags`：标签字符串数组（与 Tag 逻辑关联）。
- `description`：文件描述 / 备注说明，仅用于展示与说明，不参与权限控制。
- `author_id`：关联用户 ID。
- `visibility`：可见性，`private` / `public`，默认 `private`。
- `publicFrom` / `publicTo`：公开时间窗口（可选）。
- `publicBlocked`：是否被管理员禁止公开，默认 `false`。
- `publicBlockedAt` / `publicBlockedBy`：禁止公开的时间与操作者（管理员）。
- `created_at` / `updated_at`：创建与更新时间。

- **Note**
  - 与 Document 类似，增加 `priority` 字段（优先级）。

- **Tag**
  - `name`：标签名称。
  - `color`：颜色值。
  - `owner_id`：拥有者用户 ID。

- **Category**
  - `name`：分类名称。
  - `slug`：唯一标识。
  - `description`：描述。
  - `icon` / `color`：展示用字段。
  - `owner_id`：拥有者用户 ID。

- **Index**（向量索引，占位实现）
  - `documentId`：关联的文档 ID。
  - `vector`：文档内容的向量表示（当前为伪向量）。
  - `metadata`：包含 `title`、`tags`、`category`、`userId` 等。

- **SearchLog**（搜索日志）
  - `userId`：发起搜索的用户 ID。
  - `query`：搜索关键字 / 问句。
  - `createdAt`：搜索时间。

- **DocumentShare**（文档定向共享）
  - `documentId`：被共享的文档 ID。
  - `ownerId`：发起共享的用户 ID（一般为文档作者）。
  - `targetUserId`：被共享的目标用户 ID。
  - `expiresAt`：共享失效时间（最大为创建时间 + 30 天）。
  - `createdAt`：共享创建时间。

### 五、功能需求与后端接口映射

> 本节只做“模块级”说明，接口细节参考 `API_DOCUMENTATION.md`。

#### 5.1 用户管理

- **需求**：
  - 用户可以通过邮箱 + 密码注册、登录、登出。
  - 登录成功后获取 JWT，用于访问其他受保护资源。
  - 所有与知识库相关的数据必须与用户 ID 绑定，并通过 JWT 中的用户信息进行访问控制。
  - 支持基础角色体系与权限控制：
    - `viewer`：只读用户，只能查看自己有访问权限的内容，不能创建/编辑/删除文档、笔记、标签、分类等资源。
    - `user`：普通用户，可以在自己的空间内创建/编辑/删除文档与笔记，管理标签与分类。
    - `admin`：管理员，在拥有 `user` 能力基础上，可查看与管理全局资源（例如处理权限申请、违规内容等）。

- **接口与实现**：
  - 接口：见 `API_DOCUMENTATION.md` 中「用户认证接口」。
  - 实现：`authController` + `User` 模型 + `middleware/auth`。
  - 满足点：
    - 邮箱注册 / 登录 ✅
    - 个人知识库隔离 ✅
    - 多角色权限控制（viewer / user / admin）✅

- **扩展需求：头像与密码管理**：
  - 注册时可选上传头像:
    - 接口: `POST /auth/register`
    - 行为:
      - 支持同时上传文本字段（email/username/password）与头像文件 `avatar`。
      - 头像限制: `jpg` / `png` / `webp`，大小不超过 2MB。
      - 服务端存储文件并返回可访问的 `avatarUrl`（如 `/static/avatars/xxx.png`）。
  - 用户个人资料维护:
    - 接口: `PUT /auth/profile`
    - 行为:
      - 支持在一次请求中同时修改密码与头像，或仅修改其中一项。
      - 修改密码时必须提供 `currentPassword` + `newPassword`，并校验当前密码正确性。
      - 上传新头像时，需满足同注册头像的格式与大小限制。
      - 更新成功后返回最新的用户基础信息（含 `avatarUrl`），供前端刷新展示。

- **扩展需求：权限申请与审批**：
  - 角色定义：
    - `viewer`：初始权限较低，仅支持浏览已有内容。
    - `user`：标准使用角色，可创建/编辑/删除自身资源。
    - `admin`：系统管理员，具备审批与全局管理能力。
  - 角色约束：
    - `viewer`：
      - 可调用所有只读接口：文档/笔记/标签/分类的查询、详情、搜索、图谱、统计等。
      - 不可调用任何写操作接口：禁止创建/修改/删除文档、笔记、标签、分类，也不能调整文档可见性或执行共享。
    - `user`：
      - 可在自身空间创建/编辑/删除文档与笔记，维护标签和分类。
      - 仅能管理自己创建的文档权限与共享设置。
    - `admin`：
      - 在 `user` 能力基础上，可追加全局管理能力（例如审批权限申请、未来扩展的违规处理等）。
  - 权限申请流程：
    - viewer 如需提升为 user，需在前端触发“申请权限”操作：
      - 调用 `POST /auth/role-requests`，提交目标角色（当前为 `user`）和申请理由。
      - 系统记录一条状态为 `pending` 的权限申请（`RoleRequest`）。
      - 若同一用户存在未处理的 `pending` 申请，则拒绝重复提交。
    - 管理员在“权限申请管理”页面：
      - 通过 `GET /auth/role-requests` 查看所有或按状态过滤的申请列表。
      - 对单条申请调用：
        - `POST /auth/role-requests/:id/approve`：同意申请，将目标用户角色更新为 `user`；
        - `POST /auth/role-requests/:id/reject`：拒绝申请，保留原有角色。
      - 两个接口都支持记录 `decisionComment` 作为审批意见。

#### 5.2 知识库管理（文档 / 笔记 / 标签 / 分类）

- **文档管理**：
  - 目标：提供文档从创建、浏览、搜索、编辑、删除到版本管理的完整生命周期支撑。
  - 接口：`/documents` 系列（详情见 API 文档）。
  - 功能要点：
    - `POST /documents`：创建文档（包括正文内容与文件描述 / 备注，支持后续扩展上传元数据）。
    - `POST /documents/query`：复杂查询、分页、过滤、排序，支持模式（默认/收藏/最近/已归档）与按标签、名称、上传用户过滤，仅返回列表元数据，不包含文件/正文本身。
    - `GET /documents/:id`：详情，返回文档元信息（名称、标签、分类、上传人、公开时间窗口等）以及文件下载链接 `downloadUrl`，但不直接返回文件内容；文件下载通过专门的导出接口完成。
    - `PUT /documents/:id`：轻量更新元数据（如标题、分类、标签），不建议在此接口修改文件内容。
    - `POST /documents/:id/save`：编辑保存接口，仅允许修改文件描述 / 备注，不允许修改原有文件内容。
    - `PUT /documents/:id/visibility`：设置文档可见性与公开时间窗口。
    - `DELETE /documents/:id`：删除。
    - `/documents/:id/versions...`：版本列表、查看、恢复。
  - 权限与公开规则：
    - 文档默认 `visibility = private`，仅作者本人与管理员可访问。
    - 作者可设置 `visibility = public` 并指定 `publicFrom/publicTo`，在时间窗口内文档对所有用户可见。
    - 管理员可以对公开文档进行“禁止公开”处理（`publicBlocked = true`），即使作者设置为 public 也不再对普通用户展示。
    - 作者还可以通过 `DocumentShare` 针对指定用户做“定向共享”，共享时长最大 30 天，到期自动失效。
  - 用户级视图与元数据：
    - 通过 `UserDocumentMeta` 记录用户与文档之间的关系：收藏、归档、最近访问时间等。
    - 前端“默认查询 / 收藏 / 最近 / 已归档”四个视图均基于同一文档集合，通过 `mode` 字段和 `UserDocumentMeta` 进行区分。

- **笔记管理**：
  - 与文档类似，更轻量，强调 `priority` 字段用于代办 / 想法记录。
  - 接口：`/notes` 系列。

- **标签 / 分类管理**：
  - 提供基础 CRUD 功能，供列表过滤、标签云和文件夹树使用。
  - 接口：`/tags`、`/categories` 系列。
  - 后续可扩展为树形层级、空间级别的分类体系。

#### 5.5 viewer 权限与文档预览控制（本次新增的完整需求）

- **5.5.1 角色说明补充**
  - `admin`：
    - 拥有系统的最高权限，可查看和管理所有用户（user、viewer）的信息与行为记录。
    - 可调整 viewer 权限（修改默认每日可预览次数、临时或永久升级为 user）、审批 viewer 的预览配额申请、管理冷静期。
  - `user`：
    - 正常使用角色，拥有当前系统设计下与知识库相关的所有业务功能：
      - 文档/笔记的创建、编辑、删除、收藏、归档、导出。
      - 标签和分类的创建、编辑、删除。
      - 文档权限设置（公开窗口、定向共享）。
      - 使用搜索、语义搜索、统计、知识图谱等能力。
  - `viewer`：
    - 仅允许读取的受限角色：
      - 可以查看自己有权限范围内的文档列表、基本元数据（标题、标签、分类、公开时间窗口等）。
      - 不允许执行任何写操作：包括文档/笔记/标签/分类的创建、编辑、删除以及权限设置、共享等。
      - 预览（查看文档内容）受到严格配额控制，详见下文。

- **5.5.2 viewer 文档预览规则**
  - 默认每日可预览文件数：
    - 每个 viewer 账户都有一个“每日默认可预览文件数”（`viewerDailyLimit`），例如默认 10。
    - 该配置为**以用户为粒度**，可以由 admin 单独调整，不影响其他 viewer。
  - 24 小时计数规则：
    - 对于某个 viewer 与某个文档的组合：
      - 在 24 小时内首次成功预览该文档时，记为“预览 1 次”，消耗 1 次配额，并记录“首次预览时间”。
      - 之后 24 小时内再次打开同一文档内容不再额外计数，但可记录访问日志以备审计。
    - 当日已被计数的“不同文档数量”达到当日配额（基础额度 + 当前生效的额外额度）后，该 viewer 当天不能再预览新的文档内容。
  - 预览行为日志（ViewerPreviewLog）：
    - 每次“首次成功预览”的动作需要写入一条日志记录：
      - `viewerId`：viewer 用户 ID。
      - `documentId`：文档 ID。
      - `firstViewedAt`：首次成功预览的时间戳。
      - `source`：配额来源（例如 `default` / `extra`），区分消耗的是默认配额还是管理员额外授权配额。
    - 后续同一 viewer 在 24 小时内再次打开同一文档时，可选记录“访问事件”用于更细粒度追踪，但不影响配额计数。

- **5.5.3 viewer 申请新增预览次数**
  - 申请触发条件：
    - 当某个 viewer 在当前统计周期内已用尽可用配额（包括默认 `viewerDailyLimit` 与当前生效的额外配额），尝试预览新的文档时：
      - 后端返回“配额已用完”的业务错误，并提示前端可发起“申请新增预览次数”。
      - viewer 主动点击“申请提高预览次数”后，才会创建一条新的配额申请记录。
  - 申请记录模型（PreviewQuotaRequest）：
    - 字段示例：
      - `viewerId`：申请人（viewer 用户 ID）。
      - `status`：`pending` / `approved` / `rejected` / `auto_rejected`。
      - `requestedExtraTotal`：本次希望增加的配额数量（可由前端传入建议值）。
      - `extraTotal` / `extraRemaining`：管理员最终批准的额外配额总数及剩余可用次数。
      - `effectiveFrom` / `effectiveTo`：本次额外配额的生效时间窗口（例如当前时间起 3 天内有效）。
      - `cooldownUntil`：冷静期结束时间（详见下文）。
      - `createdAt`：申请创建时间。
      - `decidedAt` / `decidedBy`：审批时间与审批管理员 ID。
      - `decisionComment`：审批意见（例如“临时增加 10 次配额，有效期 3 天”）。
      - `meta`（可选）：如申请来源页面、备注字段等。
  - 申请自动拒绝：
    - 若申请状态在创建后 24 小时内仍为 `pending`（管理员未处理）：
      - 系统自动将状态更新为 `auto_rejected`，并记录 `decidedAt` 与统一的拒绝说明。
      - 在前端和管理后台中对 `auto_rejected` 类型的拒绝进行特殊标识，方便区分人为拒绝与系统超时拒绝。
  - 申请审批通过后的行为：
    - 管理员在审批时可设置：
      - 生效时间窗口：例如默认 24 小时、或“3 天内有效”等。
      - 额外可预览文件总数：例如“在该生效窗口内额外增加 10 次预览配额”（对应 10 个“24 小时内首次预览计数”）。
    - 生效窗口内：
      - viewer 可以在日常使用中优先消耗这次额外配额（`extraRemaining` 随使用递减）。
      - 在 `effectiveFrom` ~ `effectiveTo` 时间段内，无论 `extraRemaining` 是否已经用完，viewer 均**不能再次发起新的配额申请**（避免频繁申请轰炸管理员）。
  - 冷静期（cooldown）：
    - 当 `effectiveTo` 到达后，系统自动进入一个**冷静期 12 小时**：
      - 在冷静期内，viewer 不能再发起新的配额申请。
      - 管理员可以在管理后台中：
        - 手动提前结束冷静期，使该 viewer 重新具备申请资格。
        - 手动延长冷静期，以限制高频使用者的申请频率。

- **5.5.4 管理员对 viewer 的管理能力**
  - 查看用户信息：
    - 管理员可通过管理接口查看所有 `user` 和 `viewer` 的基本信息列表（用户名、邮箱、角色、创建时间等），并支持按角色、关键字过滤。
    - 查看单个 user 详情时：
      - 需返回该 user 的文档提交/修改记录表（DocumentActivityLog），并支持对每一条记录下载对应版本的文件内容。
    - 查看单个 viewer 详情时：
      - 需返回该 viewer 的预览行为日志（ViewerPreviewLog）、配额申请记录（PreviewQuotaRequest），以及当前生效的额外配额与冷静期配置。
  - 调整 viewer 默认每日配额：
    - 管理员可在“编辑 viewer 信息”界面中修改字段 `viewerDailyLimit`，例如：
      - 从默认的 10 次/天调整为 20 次/天。
      - 将某个高风险 viewer 的每日配额降低为 0。
    - 此类修改为**长期生效**（“永久”），不会自动回退，后续如需再次调整必须由管理员主动操作。
  - 临时/长期升级 viewer 为 user：
    - 管理员可提供两种升级模式：
      - 临时升级：为 viewer 设置一个 `temporaryRoleUntil` 字段，在该时间之前以 `user` 身份使用系统，到期自动恢复为 `viewer`。
      - 永久升级：直接将 `role` 字段改为 `user`，不再自动恢复为 `viewer`。
    - 具体的升级接口和字段约定将通过管理接口在 `API_DOCUMENTATION.md` 中单独定义。
  - 对配额申请的处理：
    - 管理员可以通过审批接口对单条 `PreviewQuotaRequest` 进行“同意 / 拒绝”操作：
      - 同意：填写生效时间（默认 24 小时，可配置）和额外可预览文件数；系统自动更新 `extraTotal/extraRemaining`、`effectiveFrom/effectiveTo` 以及 `cooldownUntil`。
      - 拒绝：更新状态为 `rejected`，可填写拒绝原因；系统可选择是否进入冷静期（推荐：拒绝本身不自动触发冷静期，仅靠通过后的冷静期机制限制频率）。

#### 5.3 AI 增强功能

- **语义搜索 / 智能问答**：
  - 接口：`POST /search`
  - 功能：
    - 对用户输入的自然语言问题，在其私有知识库中检索相关文档。
    - 通过向量相似度排序，返回文档列表和一个“答案片段”。
  - 当前状态：
    - 使用 `fakeEmbed` + 点积作为占位实现。
    - 接口契约与真实 RAG 流程兼容，可后续替换为 OpenAI / 本地模型实现。
  - 搜索日志：
    - 每次调用语义搜索接口都会在 `SearchLog` 中记录一条日志，用于统计用户搜索行为和构建“最近 3/7/30 天搜索情况”等报表。

- **向量索引维护**：
  - 服务：`aiService.upsertDocumentIndex(doc)`。
  - 需求：在文档创建 / 更新时维护 Index 记录，未来可增加批量重建接口。

- **知识图谱**：
  - 接口：`GET /graph`
  - 功能：
    - 将文档、标签、分类转换为节点（document/tag/category）。
    - 通过 `belongs_to`、`has_tag` 等边描述关系。
    - 前端基于此数据渲染知识图谱。
  - 未来扩展：
    - 内容级实体抽取与关系识别，形成更丰富的语义图谱。

#### 5.4 数据可视化与统计

- **概览统计**：
  - 接口：`GET /analytics/overview`
  - 功能：返回文档总数、笔记总数等基础指标。

- **使用趋势**：
  - 接口：`GET /analytics/trends`
  - 功能：按日期聚合文档 / 笔记创建数量，用于折线图/柱状图。

- **用户级统计**：
  - 接口：`GET /analytics/user-summary`
  - 功能：
    - 返回当前用户的文档数、笔记数。
    - 返回最近 3 / 7 / 30 天内的文档变更次数（按 `updated_at` 统计）。
    - 返回最近 3 / 7 / 30 天内的语义搜索次数（基于 `SearchLog.createdAt` 统计）。

### 六、非功能需求（NFR）

#### 6.1 性能与容量

- 单次请求响应时间：
  - 常规接口（认证、详情、列表）：在正常数据量下 P95 < 500ms。
- 并发能力：
  - 单实例支持至少 100 并发用户（Node.js + 合理连接池配置）。
- 数据规模假设：
  - 单用户 1,000~5,000 篇文档，10,000 条笔记以内。
  - 初期可通过索引优化与分页控制满足需求。

#### 6.2 安全与合规

- 认证与授权：
  - 所有非公开接口必须携带合法 JWT。
  - 所有数据访问必须通过 `author_id/owner_id` 做过滤，防止跨用户访问。
- 数据保护：
  - 密码必须做哈希存储，不允许明文。
  - 敏感配置通过环境变量注入（`JWT_SECRET`、`MONGODB_URI`）。
- 传输安全：
  - 正式环境必须部署在 HTTPS 之上。

#### 6.3 可观测性（日志 & 监控）

- 日志：
  - 统一使用中间件打印基础访问日志（method、url、status、response time）。
  - 错误由全局错误中间件记录（附 traceId，方便排查）。
- 健康检查：
  - `GET /health` 返回服务状态、当前环境、API 前缀信息。

#### 6.4 可维护性与扩展性

- 架构分层：
  - `config/`：环境与应用配置。
  - `app.js`：应用实例与中间件注册。
  - `server.js`：进程启动入口。
  - `routes/`：路由层，按领域拆分。
  - `controllers/`：控制器层，只处理请求与业务编排。
  - `models/`：数据模型层（Mongoose）。
  - `middleware/`：横切逻辑（认证、错误处理、异步包装等）。
  - `services/`：领域服务（AI、索引维护等）。
- 版本演进：
  - 通过 `config.apiVersion` 和多前缀挂载策略，实现平滑升级。

### 七、错误码与返回结构约定（简要）

- **返回结构**：
  - 统一为：`success / code / message / data / meta`。
  - 其中 `meta` 至少包含：`timestamp`、`traceId`。

- **错误码示例**：
  - 40001：参数验证失败（ValidationError）。
  - 40002：数据库唯一约束冲突（如字段已存在）。
  - 40004：资源已存在（如用户已存在、标签已存在）。
  - 40100：未授权，缺少访问令牌。
  - 40101：未授权，访问令牌无效或已过期。
  - 50000：服务器内部错误（未分类）。

实际错误码列表可在后续需求中细化并追加表格形式展示。

### 七-B、最近功能更新（2026-03）

本节记录最近对系统功能的重要更新和增强。

#### 1. 管理员权限增强

- **全局数据可见性**：
  - 管理员现在可以查看**所有用户的知识库数据**，包括：
    - 文档列表（`POST /documents/query`）
    - 笔记列表（`POST /notes/query`）
    - 知识图谱（`GET /graph`）
    - 统计分析（`GET /analytics/*`）
  - 普通用户仍然只能查看自己的数据。

- **文档可见性管理**：
  - 管理员可以修改任意文档的可见性设置（`PUT /documents/:id/visibility`）。
  - 当管理员修改他人文档的可见性时，系统会自动向原作者发送通知。

#### 2. 通知系统

- **新增 Notification 模型**：
  - 支持系统内消息推送。
  - 当前支持的通知类型：`visibility_changed`（可见性变更通知）。

- **通知接口**：
  - `POST /notifications/query`：查询通知列表。
  - `POST /notifications/:id/read`：标记通知为已读。

#### 3. 文档接口增强

- **返回字段增强**：
  - 所有文档相关接口现在返回：
    - `uploader`：上传人信息（`id`、`username`、`email`、`avatarUrl`）。
    - `updated_at`：最后更新时间。
    - `canEdit`：当前用户是否可编辑（作者或管理员为 `true`）。

- **编辑保存支持内容替换**：
  - `POST /documents/:id/save` 现在支持通过 `content` 参数替换文档内容。
  - 实现"删除原有文件上传新的文件"的需求。

#### 4. 标签管理优化

- **查询接口改为 POST**：
  - 新增 `POST /tags/query`，支持分页和关键字搜索。
  - 保留 `GET /tags` 作为兼容接口。

- **权限调整**：
  - 管理员可看到所有标签。
  - 普通用户只能看到自己的标签 + 全局标签（`is_global = true`）。

#### 5. 分类/文件夹私有化

- **完全私有模式**：
  - 分类（文件夹）现在为**完全私有**，每个用户只能看到自己创建的分类。
  - **管理员也无法看到其他用户的私有文件夹**。
  - 只有全局分类（`is_global = true`）对所有用户可见。

- **权限限制**：
  - 分类的更新和删除只能由创建者本人操作。
  - 即使是管理员也不能修改或删除他人的私有分类。

#### 6. 系统默认文件夹

- **虚拟文件夹**：
  - 系统默认提供3个虚拟文件夹，基于 `UserDocumentMeta` 动态计算：
    - **收藏**（`favorite`）：`isFavorite = true` 的文档。
    - **最近**（`recent`）：`lastAccessedAt` 不为空的文档。
    - **已归档**（`archived`）：`isArchived = true` 的文档。

- **自动行为**：
  - 新上传的文件自动进入「最近」（`lastAccessedAt` 设为当前时间）。
  - 30天内无改动的文件自动进入「已归档」。
  - **收藏的文件不会被自动归档**。

- **手动触发归档**：
  - `POST /categories/auto-archive`：手动触发自动归档检查。

#### 7. 数据模型变更

- **Category 模型**：
  - 新增 `is_global` 字段，支持全局分类。

- **Tag 模型**：
  - 新增 `is_global` 字段，支持全局标签。

- **Notification 模型**（新增）：
  - `userId`：通知接收者。
  - `type`：通知类型。
  - `title`、`body`：通知内容。
  - `relatedDocumentId`、`relatedUserId`：关联的文档和用户。
  - `read`：是否已读。
